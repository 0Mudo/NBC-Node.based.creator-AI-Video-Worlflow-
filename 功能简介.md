## 1. 一句话概述（One-liner）

NBC 是一款**节点式 AI 素材创作工作流桌面应用**：用户通过拖拽连线组合「角色/场景/物品/剧本/分镜/提示词」等节点，并驱动「GPT Image 2 / Seedance / Banana」等生成节点批量产出图像与视频；生成结果进入素材库与时间线，最终导出视频或脚本。

## 2. 背景与问题（Background & Problem）

### 2.1 背景（从产品视角归纳）
- AI 图像/视频创作通常分散在多工具、多网站、多份 prompt 文档中，导致复用难、版本混乱、素材难回溯。
- 从“灵感 → 剧本 → 分镜 → 生成 → 归档 → 拼接导出”链路长，手工复制粘贴成本高。

### 2.2 主要问题（用户痛点）
- 痛点 P1：prompt/参数缺乏结构化管理，上游设定（角色/场景）难以稳定注入到生成请求。
- 痛点 P2：生成结果散落在不同目录/平台，难以按项目聚合、打标、复用和填充到分镜。
- 痛点 P3：多模型/多端点 API Key 管理困难（环境切换、密钥泄露、CORS 等）。

### 2.3 目标（Goals）
- G1：把创作流程拆成可复用“节点”并形成可视化数据流，降低 prompt 拼装与批量生成成本。
- G2：把“生成结果”统一沉淀为可索引素材（本地/OSS/飞书），并能直接拖拽复用到工作流与时间线。
- G3：提供可操作的质量反馈（失败日志/生成分析/提示词优化）以提升成功率与一致性。

### 2.4 非目标（Non-goals / 明确边界）
- NG1：不是云端协作产品：**无服务端、无多用户实时协作**；数据以本地 localStorage/本地文件为主。
- NG2：不承诺对任意外部模型/平台通用适配；当前主路径围绕：GPT Image 2、Seedance、Banana。
- NG3：不在本阶段解决“账户体系/权限体系/计费/配额”等平台化问题。

## 3. 目标用户与使用场景（Users & Scenarios）

### 3.1 目标用户（Personas）
- U1：独立创作者/视频剪辑/导演助理  
  - 目的：快速把灵感变成一组可用镜头素材，并导出成片或交付素材包。  
  - 高频任务：写剧本/分镜、调整 prompt、批量生成、挑选素材、时间线拼接导出。  
  - 约束：时间紧、迭代频繁、素材归档要求明确（按项目）。  

- U2：团队中的“素材生产”角色（负责批量出图/出视频）  
  - 目的：高效批量生成、降低失败率、把结果同步到团队资产库（OSS/飞书）。  
  - 高频任务：按模板跑工作流、监控队列、处理失败、同步/打标/分发。  

### 3.2 典型场景（Top Scenarios）
- S1：从文本设定生成一组角色/场景一致的镜头图/视频素材（节点工作流）。
- S2：把已有素材（本地/OSS/飞书）拖入工作流作为参考，做图生图/图生视频迭代。
- S3：把生成素材放入时间线轨道，生成导出脚本或导出视频（Electron + ffmpeg concat）。
- S4：在灵感编辑器里用对话式迭代改剧本/分镜，并把结构化信息映射到“卡片/节点”。

## 4. 核心体验（Core Experience）

### 4.1 核心价值（Value Proposition）
- 对用户：把创意生产拆解为可视化组件（节点+面板），降低重复劳动，提升复用与可追溯性。
- 对团队：把素材/配置/事件记录结构化沉淀，便于归档、共享、回放与自动同步。

### 4.2 顶层用户流程（Happy Path）
> 用「输入 → 处理 → 输出」表达，便于 agent 抽取与自动化。

1. 进入应用 →（首次）完成 Provider 配置向导/设置  
2. 创建/切换项目（Project）  
3. 在画布中拖拽节点、连线形成工作流（Workflow Graph）  
4. 编辑节点参数（Inspector）  
5. 运行单节点或“全部运行”（执行引擎：拓扑排序→收集上游→解析模板→调用外部 API）  
6. 结果进入：生成队列/失败日志/通知面板；生成素材写入素材库（Asset）  
7. 在时间线拖拽素材填充镜头轨道 →（可选）导出脚本/导出视频  
8.（可选）输出节点三路保存（本地/OSS/飞书队列+飞书云盘）

## 5. 功能范围（Scope / 以代码为准）

> 每条能力都提供：定位、关键入口、关联代码路径（相对仓库根目录）。

### 5.1 节点式工作流（Node Graph）
- F1：节点画布编辑（ReactFlow）  
  - 能力：拖拽节点、连线、选择/删除、多选、导入导出文件、全部运行  
  - 入口：`src/components/node-editor/FlowEditor.tsx`  
  - 数据：`useFlowStore`（nodes/edges）、`useProjectStore`（持久化）

- F2：节点类型（以 `src/nodes/index.ts` 为准，共 11 种）  
  - 输入/设定：`assetInput`（素材输入）、`characterCard`（角色卡）、`sceneCard`（场景卡）、`itemCard`（物品卡）、`script`（剧本）、`storyboard`（分镜）  
  - 处理：`prompt`（提示词，支持模板变量）  
  - 生成：`gptImage2`（图像）、`seedance`（视频）、`banana`（图像）  
  - 输出：`output`（三路保存策略）  

- F3：工作流执行引擎  
  - 能力：拓扑排序、收集上游文本/参考图、并发批量生成、任务队列与取消、写失败日志、触发事件与通知  
  - 入口：`src/store/useExecutionEngine.ts`、`src/engine/graph.ts`、`src/engine/promptResolver.ts`  

### 5.2 灵感编辑器（剧本/分镜/三卡）
- F4：灵感编辑器与结构化卡片编辑  
  - 能力（来自代码结构与手册描述）：剧本→分镜→三卡的编辑与迭代；对话式辅助；版本/历史管理（具体字段以 store 为准）  
  - 入口：`src/components/inspiration-editor/InspirationEditor.tsx`、`src/store/useInspirationStore.ts`

### 5.3 素材库（Asset Browser）
- F5：素材浏览器（本地 + OSS + 飞书云盘 + 预设）  
  - 能力：按标签过滤、按项目索引、拖拽到画布/时间线、详情查看、标签编辑、删除  
  - 入口：`src/components/asset-browser/*`、`src/store/useAssetStore.ts`  
  - 数据模型：`src/types/asset.ts`（Asset / tags / source）

### 5.4 时间线分镜系统（Timeline）
- F6：时间线轨道与镜头片段（tracks/clips）  
  - 能力：轨道（视频/对白/字幕/BGM 等）、从脚本导入镜头、拖拽素材、保存与迁移旧数据结构  
  - 入口：`src/components/timeline/TimelineView.tsx`、`src/store/useTimelineStore.ts`  
  - 持久化：`localStorage` key `nbc_timeline_v2_<projectId>`（含 legacy 迁移）

### 5.5 生成队列、失败日志、生成分析、提示词库
- F7：生成队列 / 失败日志  
  - 入口：`src/components/generation-queue/*`、`src/store/useGenerationStore.ts`、`src/store/useLogStore.ts`  
  - 数据模型：`src/types/generation.ts`（GenerationTask）

- F8：生成分析  
  - 入口：`src/components/analytics/AnalyticsPanel.tsx`

- F9：提示词库（Prompt Library）  
  - 入口：`src/components/prompt-library/PromptLibraryPanel.tsx`、`src/store/usePromptLibraryStore.ts`

### 5.6 Provider 配置系统（外部服务端点/密钥）
- F10：Provider 配置、启用/禁用、连通性测试、导入导出  
  - Provider registry：`src/store/useProviderStore.ts`（默认 providers + localStorage merge）  
  - 默认 Provider（代码现状）：`gptImage2` / `seedance` / `banana` / `llm` / `oss` / `feishuDrive`  
  - 注意：执行引擎会按 capability 选择“已启用且有 apiKey 的 provider”，否则退回默认 id。

### 5.7 三路保存（本地 / OSS / 飞书）
- F11：输出节点 + 保存管理器  
  - 入口：`src/api/saveManager.ts`、`electron/main.ts`（IPC：`save:local` / `save:oss` / `save:feishu` 等）  
  - 行为：下载结果（base64/blob）→ 本地保存（Electron）→ OSS 上传（Electron）→ 飞书同步队列 & 可选飞书云盘直传（Electron）

### 5.8 导出（Export）
- F12：导出视频（Electron）  
  - UI：`src/components/export/ExportPanel.tsx`  
  - IPC：`electron/preload.ts` → `ipcRenderer.invoke('export:video', …)`  
  - 主进程实现：`electron/main.ts`（使用 `ffmpeg -f concat -c copy`；若 ffmpeg 不可用则输出拼接 list 文件）  
  - 重要限制：当前实现主要面向**本地文件路径**；远程 URL 若未落盘则无法拼接。

## 6. 业务规则与约束（Rules & Constraints）

### 6.1 关键规则（以代码逻辑归纳）
- R1：执行顺序由拓扑排序确定（`src/engine/graph.ts topoSort`）。
- R2：提示词会从上游节点收集并进行模板解析（`src/engine/promptResolver.ts`；支持 negative prompt、参考图收集、seed 等）。
- R3：生成节点支持 `batchCount` 并发批量（`src/store/useExecutionEngine.ts`）。
- R4：Provider 选择规则：优先选择“能力匹配 + enabled + endpoint 有 apiKey”，否则 fallback 到默认 provider id（`getProviderForType`）。
- R5：关键用户操作会发事件（`emitNBCEvent`），Electron 环境写入 JSONL，Web 环境写入 localStorage 队列（`src/utils/nbcEvents.ts`）。

### 6.2 运行约束（Constraints）
- C1：无后端：所有状态保存在本地（localStorage + 本地目录 + 导入导出文件）。
- C2：外部 API 可用性不由 NBC 保证；NBC 需要对失败进行记录、重试/取消（当前代码：失败日志+取消）。
- C3：Electron 与 Web 模式能力不同：  
  - Electron：可访问文件系统、可 IPC 代理请求（避免 CORS）、可导出视频/上传 OSS/飞书。  
  - Web：文件能力受限，部分功能为“模拟/降级”。

## 7. 数据与对象模型（Data / Entities）

> 这里给出 agent 可直接用于建模/测试的最小对象集合（字段名以 TypeScript 类型为准）。

### 7.1 核心对象（Entities）
- E1：Project（项目）  
  - 类型：`src/types/project.ts Project`  
  - 字段：`id, name, createdAt, updatedAt, nodeCount`

- E2：ProjectData（项目图数据）  
  - 类型：`src/types/project.ts ProjectData`  
  - 字段：`nodes: AppNode[], edges: AppEdge[]`

- E3：Workflow Graph Node（节点）  
  - 类型：`src/types/flow.ts AppNode<NodeData, NodeType>`  
  - 关键字段：`id, type, data(label + 各节点参数), position`

- E4：ProviderConfig（外部服务配置）  
  - 类型：`src/types/provider.ts ProviderConfig`  
  - 字段：`id, name, capabilities, endpoints[{url, apiKey, isDefault,…}], enabled`

- E5：Asset（素材）  
  - 类型：`src/types/asset.ts Asset`  
  - 字段：`id, name, type(image|video|panorama|text), path, tags[], createdAt, projectId?, source?`

- E6：Timeline（时间线 v2）  
  - 类型：`src/store/useTimelineStore.ts Track / TrackClip`  
  - 关键字段：`tracks[]`；clip：`startTime, duration, sourceUrl, sourceType, spec{characterIds, sceneId, shotType, transition, dialogue…}`

- E7：GenerationTask（生成任务）  
  - 类型：`src/types/generation.ts GenerationTask`  
  - 字段：`id, nodeId, type(gptImage2|seedance|banana), status, progress, resultUrl?, error?`

### 7.2 持久化（localStorage keys，便于 agent/debug）
- `nbc_projects`：项目列表  
- `nbc_active_project`：当前项目  
- `nbc_project_<id>`：项目 nodes/edges  
- `nbc_workflows`：工作流库（独立于项目）  
- `nbc_providers`：Provider registry  
- `nbc_failure_reports`：失败日志  
- `nbc_operator_name`：操作者名称（用于事件/同步记录）  
- `nbc_event_queue`：Web fallback 事件队列  
- `nbc_timeline_v2_<projectId>`：时间线 v2  
- `nbc_timeline_<projectId>`：时间线 legacy（会被迁移）

### 7.3 导入导出文件（`.nbc.json`）
- 文件结构：`src/types/project.ts NbcFile`  
  - 字段：`version, projectName, nodes, edges, metadata{ timeline?, inspiration?, appVersion… }`

## 10. 质量属性（Non-functional Requirements）

- 可用性：本地离线可打开与编辑项目；外部 API 不可用时应给出清晰错误（失败日志/通知）。
- 安全：Electron 模式通过 IPC 代理外部请求，避免在渲染进程直接暴露密钥；`contextIsolation: true`（详见 `AGENTS.md`）。
- 可观测性：关键动作发事件（JSONL/队列）；失败日志可复盘。
- 可维护性：节点类型与 store/类型分离；Provider registry 可扩展（add/remove provider）。

---

## 11. 验收标准（Acceptance Criteria）

- AC1（项目）：可创建/切换项目；项目 nodes/edges 能持久化并重新载入（`nbc_project_<id>`）。
- AC2（节点/执行）：至少支持 11 种节点类型；生成节点可运行并产生 `resultUrl`，失败会写入失败日志。
- AC3（素材）：生成结果会进入素材库，可按标签过滤，并可拖拽到画布/时间线。
- AC4（时间线）：可从脚本文本导入镜头到时间线轨道；时间线数据能按项目持久化。
- AC5（导出）：Electron 环境下可导出（或至少生成 concat list 文件）；失败时提示 ffmpeg 不可用等原因。
- AC6（保存）：输出节点可选择本地/OSS/飞书通道；任一通道失败不应影响其他通道结果返回，并在 errors 中体现。

## 12. 风险与待确认（Risks & Open Questions）

### 12.1 风险（Risks）
- RK1：外部模型/平台接口变更导致不可用（endpoint、参数、鉴权、配额等）。
- RK2：本地化存储（localStorage）容量与可迁移性风险（大项目/大量素材元数据）。
- RK3：视频导出依赖 ffmpeg 与本地文件路径；远程素材无法直接拼接。
- RK4：文档与代码版本不一致（例如 ComfyUI），导致 agent 误判功能可用性。

### 12.2 待确认问题（Open Questions）
- Q1：素材“落盘策略”是否需要统一（例如远程 URL 是否自动下载缓存到本地，以支持导出）？
- Q2：输出节点保存后，Asset 的 tags/projectId 归属规则是否需要强约束？
- Q3：Provider 插件体系是否要进入“用户可自定义节点类型”的产品路线？
